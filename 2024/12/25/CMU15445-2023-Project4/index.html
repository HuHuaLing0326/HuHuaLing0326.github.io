CMU15445-2023-Project4

前言

和spring不同，2023 fall 的PROJECT #4 - CONCURRENCY CONTROL是实现一个基于时间戳的MVCC协议。共有 4 个 task + 2 个选做的 bonus task，全部完成后可以实现快照隔离（snapshot isolation）和序列隔离（serializable isolation）。

MVCC 的核心在于读操作是无锁的。

Motivation 是「我们为什么需要读最新的数据？只要读的数据是一致的就可以了。」比如 OLAP 的时候，我们在 23 点开始分析购物网站的流水，只要账对的上，我们可以获得一个 23 点的 snapshot 算流水，而不需要上大量的锁得到最新的流水。

MVCC 的基本逻辑是，事务开始的时候记录一个时间戳(read_ts), 所有的读只能读到 read_ts 之前的版本，相当于在开始的时候做了一个snapshot。在commit的时候，如果发现写集在read_ts后被修改了就Abort。否则就实际写入然后将时间戳更新为commit_ts。read_ts, commit_ts 共享一套全局时间戳，每 commit 一个 txn 就自增一次。

这一章的实验会大改 Project 3 完成的算子。然而，我们在完成对算子的修改后，必须确保运行 Project 3 的测试代码不存在报错

Task 1

Task 1 共两个小任务，计算读和写的timestamp以及watermark。

TimeStamp

在事务（Transaction）开始时，对事务添加一个读取（read stamp）时间戳，其值为最新的事务提交时间戳。当一个版本的时间戳小于等于事务的读取时间戳时，代表该版本对于当前事务是可读的。

在事务提交时，为其分配一个提交时间戳。提交时间戳是一个单调递增的逻辑计数器，每当一个事务提交则自增1。

WaterMark

Watermark是指所有处于正在处理（in-progress）状态事务的读取时间戳的最小值，在添加一个事务时，将新添加的时间戳存储到哈希表中，在移除一个事务时，若要移除的时间戳等于watermark，要更新watermark。

Task 2

从这里开始我们就需要弄清楚 BusTub 是怎样存历史数据的了。BusTub 对 MVCC 做了简化，并不会将历史记录持久化到磁盘上，数据会存储在三个位置中：

  • Table_heap
    • 最新的数据总是在table_heap中；
  • TransanctionManager
    • 存储了table_heap中元组上一次修改的指针；
    • 可以通过 GetUndoLink 函数，根据元组获得其对应的第一个 undo_link
  • Transaction: 存储了该事务修改过的元组的历史版本（Undo_log）。
    • 每一个 Transaction 中包含了多个 undo_log
    • undo_log 中包含了一个 undo_link，代表这个 undolog 所修改的元组在之前的哪一个事务中，并给出了在之前事务中寻找的该元组的值的索引
    • undo_log 中还包含了 modified_fields_ 属性，即元组中的列是否被修改的标记
    • undo_log 中还包含了 tuple_ 属性，表示被修改的属性在之前的版本中应该是什么值。此处注意 undo_log 里的 tuple 仅包含被改动的值。

Sequential Scan

对 Sequential Scan 算子，我们要分情况对元组进行讨论。迭代器依次增加，遍历所有的元组，对每一个元组有如下操作：

  1. 如果元组对当前事务可读（元组时间戳小于等于事务时间戳，或者等于事务id），则直接跳转第三步
  2. 如果元组对当前事务不可读，则进行元组恢复
    1. 收集全部时间戳大于该事务的 undo_log 和一个时间戳小于等于该事物的undo_log
    2. 将结果输入 Tuple Reconstruction 函数，得到重建结果
  3. 判断元组是否满足过滤条件
  4. 若该元组不满足返回条件，迭代器增加，对下一个元组进行上述操作
  5. 若该元组满足返回条件，迭代器增加，返回该元组
